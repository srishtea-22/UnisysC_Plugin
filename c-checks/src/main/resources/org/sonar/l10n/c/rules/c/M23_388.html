Why is this an issue?

Global variables introduce several significant problems in software development:

  • Hidden Dependencies: Functions that rely on global state have implicit dependencies that are not visible in their signatures. This makes code harder to understand and maintain.
  • Difficult Testing: Global state persists across test cases, making it difficult to write isolated unit tests and can cause tests to have unexpected interactions.
  • Concurrency Issues: In multi-threaded programs, global variables can lead to race conditions and data corruption if not properly synchronized.
  • Reduced Modularity: Global variables tightly couple different parts of the codebase, making it harder to reuse components or understand isolated functionality.
  • Namespace Pollution: Global variables occupy space in the global namespace, increasing the risk of naming conflicts and making code less maintainable.

/* Noncompliant: Global variable */
int globalCounter = 0;

void incrementCounter() {
    globalCounter++;  // Depends on global state
}

void resetCounter() {
    globalCounter = 0;  // Modifies global state
}

How to fix it

Replace global variables with better design patterns:

1. Use local variables and pass them as parameters:

void incrementCounter(int *counter) {
    (*counter)++;  // Explicit dependency through parameter
}

int main() {
    int counter = 0;  // Local variable
    incrementCounter(&counter);
    return 0;
}

2. Use structs to encapsulate related data:

typedef struct {
    int count;
    char name[50];
} CounterState;

void incrementCounter(CounterState *state) {
    state->count++;  // Explicit dependency
}

int main() {
    CounterState state = {0, "counter"};  // Local encapsulated state
    incrementCounter(&state);
    return 0;
}

3. Use static variables within functions for state that must persist:

int getNextId() {
    static int nextId = 1;  // Limited scope, implicit state
    return nextId++;
}

By eliminating global variables, your code becomes more modular, testable, and easier to reason about.
